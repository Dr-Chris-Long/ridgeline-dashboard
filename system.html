SUMMIT framework

Structured Universal Multi-model Methodology for Institutional Testing. SUMMIT is the domain-agnostic engine. RIDGELINE is the first cartridge. Loading a new domain = inserting rows, not writing code.

The core insight: Every regulated industry has the same pattern. A corpus of rules governs what a document must contain. Someone produces the document. An adversary challenges compliance. The cost of failure is significant. SUMMIT abstracts that pattern into a loadable cartridge system.

Scenario: PL5 wildfire resource coordination

Preparedness Level 5 is the highest stage of national wildfire response. Multiple geographic areas are experiencing large, complex incidents. National resources are exhausted. Every allocation decision is a tradeoff: sending crews to Fire A means Fire B doesn't get them. Decisions are made under extreme time pressure with fatigued personnel. Mistakes at PL5 cost lives.

This is the same SUMMIT pattern. A corpus of rules governs how resources must be allocated. Someone makes allocation decisions. Those decisions will be reviewed by GAO, Congress, and after-action review teams. The cost of noncompliance ranges from preventable fatalities to billions in property loss.

Why this matters to Karen and USGS: BLM manages wildland fire. USGS provides fire science, predictive modeling, and post-fire assessment (BAER). Both agencies are in the room during PL5. This scenario shows SUMMIT solving a problem they both live with every fire season.

Cascade detection at PL5

This is where SUMMIT's cascade analysis becomes life-safety critical. The same 4 cascade types apply:

Cascade typeRIDGELINE (NEPA litigation)PL5 wildfire equivalent
Sequential dependencyESA survey failure collapses take determinationPulling crews from Fire A delays containment, forces evacuation escalation, triggers mutual aid cascade. If Fire A escapes, it merges with Fire B creating a complex.
Foundation attackBaseline data error undermines all dependent analysisWeather forecast error (wind shift) invalidates all fire behavior predictions. Every allocation decision based on that forecast is now wrong simultaneously.
Independent flankMultiple statutes violated independentlySimultaneous new starts across multiple GACCs while all Type 1 teams are committed. Each new start is independently unresourced. No single reallocation fixes it.

Methodologies

Core methodologies

▶Codebook Thematic Analysis (CTA)CORE
Source: Boyatzis (1998) — Transforming Qualitative Information: Thematic Analysis and Code Development
Application: Public comment processing across all NEPA projects
Pipeline: Comment intake (PS-02) → code extraction → consolidation → theme development (PS-03) → validity check → response generation
Output: 8-tab CTA workbook with 111 comments, 1,601 codes, 111 themes, 243 responses across 9 projects
Why this method: Published, peer-reviewed qualitative research methodology. Defensible under administrative review. Reproducible by any trained analyst.
▶Multi-model adversarial debateCORE
Design principle: Separate models from separate vendors argue opposing positions to prevent analytical softening. A single model evaluating both sides converges on the middle. Multi-vendor separation forces genuine adversarial tension.
Architecture: Protagonist (challenges document) → Antagonist (defends document) → R2 escalation if no consensus → Arbiter (independent judgment). RED from protagonist NEVER reaches consensus — always escalates.
Model stack: GPT-5.4 protagonist, Sonnet 4.6 antagonist, Opus 4.6 arbiter
Validated: Production run identified 8 fatal flaws in LC project, 3 overlapping with existing federal complaint, 5 novel
▶Compliance rules distillation (METH-RULES-001 v1.4)CORE
Method: Extract binding obligations from source authority documents into atomic, testable compliance rules with page citations
Agent: Agent 3B (A3B-SYSTEM-SDA-v2)
Input: 6 source authority documents, 4,713 pages
Output: 2,967 rules, each with source_doc_id, source_section, page_citation
Traceability: 99% of rules have page citations. Every rule links to the exact page it was extracted from.
Compression pipeline: 2,967 rules → 390 clusters (7.6:1) → 263 debate units via 39 families (11.2:1 total)
▶Recommendation methodology (4-layer system)CORE
Layer 1 — PROC-CAT-001 Finding classification: Each finding classified as RULE_VIOLATION (EA conflicts with binding rule), NOT_MENTIONED (EA silent on obligation), INADEQUATE (analysis present but insufficient), or COMPLIANT. Categories assigned from debate transcript, not inferred from severity.

Layer 2 — AR-FIX-FRAMEWORK-v1 Structured fix: Fix type (EA_REVISION, DR_SUPPLEMENT, NEW_ANALYSIS, AR_ADDITION, BIOP_REINITIATION, PROJECT_REDESIGN), responsible specialist, EA sections to revise, data needed, effort estimate, fix difficulty, blocks decision flag. Prioritized: CRITICAL > HIGH > MODERATE.

Layer 3 — AR-OUTPUT-SCHEMA-v1 Advisor output (25 fields per finding): Urgency (FIX_NOW through NO_ACTION), fixability, fix_anyway rationale (why fix even if defense could win), priority_score / fix_value / fix_cost, calendar time, if-not-fixed consequence, project file check (where in the AR to look), advisor options (multiple fix paths with tradeoffs), severability note, PI vulnerability score.

Layer 4 — PROC-DDT-001 Disclosure/Decision Test (guardrail): Every recommendation screened against two verb lists. Permitted (disclosure): add, document, quantify, identify, disclose, discuss, map, cite, classify, provide, describe, list, calculate, inventory, delineate, tabulate. Prohibited (decision): select, conclude, determine, find, reject, adopt, approve, deny, decide, choose, authorize, permit, allow, prohibit, require. Any prohibited verb triggers automatic rewrite. Recommendations tell planners what analysis to ADD to the record, never what decision to MAKE.

Human oversight

AI proposes analysis. Humans decide. This is not a design preference — it is a policy requirement enforced at the architectural level.

Non-negotiable boundary: RIDGELINE does not make significance determinations, select alternatives, or produce final agency decisions. It does not draft FONSI language, sign decision records, or generate content that enters the administrative record without human review. Every output is an analytical input for a human decision-maker.

The planner decision workflow

Every finding in the solicitor-ready deliverable includes three planner-controlled columns. These columns are the formal human oversight mechanism.

Planner decision
ACCEPT — finding is valid, implement the recommendation
DENY — finding is not actionable (rationale required)
REVISE — finding is partially valid, modify the recommendation (rationale required)
Planner rationale
Required on DENY and REVISE. Must explain why the finding was rejected or modified. Creates audit trail for after-action review.
Revised action
If REVISE selected, planner writes their own recommended action. This replaces the AI-generated recommendation in the working file.

How the AI helps humans review

The system is designed to make human review efficient and informed, not to replace it.

▶Project file check
What it does: For each finding, the advisor identifies where in the administrative record the gap might already be addressed. Example: "Check the Biological Assessment occupancy section" or "Review the hydrologist specialist report for watershed-scale analysis."
Why it matters: Many "missing analysis" findings exist in the project file but weren't included in the EA. The project file check tells the planner exactly where to look before accepting or denying a finding. This prevents false positives from wasting planner time.
Population: 167/695 findings have project file checks populated (24%)
▶Advisor options
What it does: For each finding, the advisor presents 2+ fix paths with different timelines and tradeoffs. Example: "Option A: Add 2 paragraphs of cumulative effects discussion (2 days, low risk). Option B: Commission new watershed analysis from specialist (3 weeks, eliminates vulnerability entirely)."
Why it matters: Gives the planner real choices, not a single directive. The planner decides based on project timeline, budget, and risk tolerance.
Population: 190/695 findings have advisor options populated (27%)
▶If-not-fixed consequence
What it does: States in plain language what happens if the planner chooses not to address this finding. Example: "If acreage discrepancy is not reconciled, plaintiff can argue the incidental take analysis is based on incorrect project scope, potentially invalidating the entire BiOp consultation."
Why it matters: Enables informed risk acceptance. A planner who denies a finding does so knowing the downstream consequence, not blindly.
Population: 190/695 findings have if-not-fixed populated (27%)
▶Disclosure/Decision Test (PROC-DDT-001)
What it does: Automated guardrail that screens every AI-generated recommendation before it reaches the planner. If a recommendation contains a decision verb (select, conclude, determine, approve, deny, require, etc.) it is automatically rewritten using only disclosure verbs (add, document, quantify, identify, disclose, etc.).
Why it matters: Even if a model hallucinates decision-level language, the guardrail catches it before it reaches the planner. The planner never sees a recommendation that crosses the decision boundary.
Example:
Rejected: "Require BLM to reinitiate consultation"
Rewritten: "Identify whether reinitiation triggers under 50 CFR 402.16 have been met and disclose the analysis in the EA"

Platform architecture

RIDGELINE is a cartridge inside the LEGION multi-model orchestration framework. Nine applications across four layers, with clear separation between session work (design, analysis) and pipeline execution (automated runs).

LEGION is the domain-agnostic orchestration framework. RIDGELINE is the first cartridge. The framework handles model routing, credential management, and pipeline staging. Additional cartridges can be built for other compliance domains without modifying the core framework.
